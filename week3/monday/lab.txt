Challenge Lab: Production Server Triage
This lab simulates a real-world incident investigation. You are not given a clean happy path. You will need to reason about what you find, correlate signals across multiple tools, and write up your findings the way a senior engineer would.

There are no screenshots of expected output. The server you connect to is unpredictable. Your job is to figure out what is happening.

Estimated time: 60–90 minutes


Before You Begin: Mandatory Environment Setup
Do not skip this step. If you do not run this script, you will be investigating an empty machine. This script creates the realistic simulation of a degraded server state.

Use your verified environment (WSL2 on Windows, an Ubuntu/Debian VM, or a cloud VM). If you are on macOS, you must use a Linux VM (like Multipass) or a Linux container with a shell, as some of these commands rely on the Linux /proc filesystem.

Setup Steps:

Open your terminal.
Create the setup file by running: nano setup_lab.sh
Copy the exact code block below and paste it into your terminal.
Save and exit (Press Ctrl+O, Enter, then Ctrl+X).

Incident Briefing
It is 09:14 on a Monday. The KijaniKiosk marketplace has been live for 72 hours. Osei sends the following message to the team channel:

“API response times are degraded. P95 latency went from ~120ms to ~480ms overnight. The Node.js app process is running. NGINX is running. But something is wrong with the system. I need you to SSH in, do a full triage, and tell me:
What is consuming resources abnormally.
Whether any processes are in a bad state.
Whether the filesystem has anything suspicious.
Whether there are any relevant log entries from the last 6 hours.
Write up your findings as a structured triage report. I want to be able to hand this to anyone on the team and they can understand what was happening.”
Open Reference Policy: You are encouraged to refer back to Pages 2–4 during this lab. Real engineers use references constantly. Building the habit of knowing where to look is part of the skill.


Investigation Objectives
Your investigation must address all four areas below. For each area, you will run multiple commands, interpret the output, and record your findings in the triage report.

Area 1: Process and Resource State
Identify what is running, what is consuming resources abnormally, and whether any processes are in unexpected states.

Commands to consider (not a complete list so use your judgment):

ps aux --sort=-%mem | head -15
ps aux --sort=-%cpu | head -15
cat /proc/meminfo
free -h
top -bn1 | head -20

# Check for zombie processes
ps aux | awk '$8=="Z" {print "ZOMBIE:", $0}'

# Check open file descriptors per process
find /proc -maxdepth 3 -name fd -type d 2>/dev/null | \
  awk -F/ '{print $3}' | grep -E '^[0-9]+$' | \
  while read pid; do
    count=$(ls /proc/$pid/fd 2>/dev/null | wc -l)
    name=$(cat /proc/$pid/comm 2>/dev/null)
    echo "$count $pid $name"
  done | sort -rn | head -10
What to record: The top 3 processes by memory and CPU. Any processes in Z (zombie) or D (uninterruptible sleep) state. Whether memory is under pressure.

Area 2: Filesystem and Disk
Identify whether disk usage is a contributing factor. Find large or unexpectedly recent files.

df -h
du -sh /var/log/* 2>/dev/null | sort -rh | head -10
find /var/log -name "*.log" -size +50M -ls 2>/dev/null
find /tmp -mmin -60 -type f 2>/dev/null
ls -lhtr /var/log/ | tail -10
What to record: Whether any partition is above 80% used. The largest log files. Any unexpectedly large or recently modified files.

Area 3: Log Analysis
Extract meaningful signals from application and system logs from the past 6 hours. You are looking for patterns, not just individual errors.

# Application log errors with frequency count
grep -E "ERROR|WARN|CRITICAL" /var/log/kijanikiosk/app.log | \
  awk '{print $4}' | sort | uniq -c | sort -rn

# Timeline of errors - when did they start?
grep "ERROR" /var/log/kijanikiosk/app.log | \
  awk '{print $1, $2}' | head -20

# System log for OOM killer or disk errors
grep -E "oom|OOM|killed process|I/O error|disk quota" /var/log/syslog | tail -20

# Authentication events - unexpected logins?
grep -E "Accepted|Failed|Invalid" /var/log/auth.log | \
  awk '{print $1,$2,$3,$9,$11}' | sort | uniq -c | sort -rn | head -20
What to record: When errors first appeared. The most common error type. Any patterns that suggest a single root cause (e.g., database connection failures clustering at a specific time).

Area 4: Network and Service State
Verify that services are bound to the expected ports and that connectivity is functioning.

ss -tlnp
ss -s
curl -o /dev/null -s -w "HTTP %{http_code} - %{time_total}s\n" http://localhost/
curl -o /dev/null -s -w "HTTP %{http_code} - %{time_total}s\n" http://localhost/api/health

# Check if any expected service is NOT listening
ss -tlnp | grep -E "80|443|3000|8080|5432"

# How many connections are in each TCP state?
ss -tan | awk '{print $1}' | sort | uniq -c | sort -rn
What to record: Which ports are bound and by which processes. HTTP response time and status code. Whether any expected service port is missing. TCP connection state distribution.


Deliverable: Triage Report
Write a structured triage report in a file called triage-report.md in the KijaniKiosk repository from Week 2. The report should be written so that any engineer on the team can read it and understand exactly what was found and what it means.

Your report must include the following sections:

# KijaniKiosk API Server - Triage Report
**Date:** [date]
**Investigated by:** [your name]
**Server:**[hostname or IP]
**Incident start (approximate):** [time from log evidence]

## Summary[2-3 sentences: what you found and whether you identified a likely root cause]

## Process and Resource State[What you found. Be specific: process names, PIDs, memory/CPU figures]

## Filesystem and Disk[Disk usage findings. Any large or unexpected files]

## Log Analysis[Key log entries. When errors started. Error frequency. Any patterns]

## Network and Service State[Port binding status. HTTP response time. TCP connection state]

## Assessment
[Your best hypothesis for the root cause of the latency increase]

## Recommended Next Steps[What should be done next. Maximum 3 actions, most important first]

Evidence to Submit
A screenshot or terminal recording showing at minimum: your process investigation commands and output, your log analysis pipeline and output, and your disk analysis output.
The triage-report.md file committed to your KijaniKiosk repository under a branch named feature/week3-monday-triage.
A pull request from that branch to develop with a clear PR description explaining what the report contains.

Senior Engineering Standard
This lab is assessed against the standard of a senior engineer’s incident response, not a checklist completion. The markers of quality for completion are:

Correlation: Did you connect findings across areas? (e.g., log errors match the memory pressure timeline)
Signal vs noise: Did you identify what is significant versus what is normal background noise?
Hypothesis quality: Is your root cause assessment defensible from the evidence you collected?
Actionability: Are your next steps specific enough for another engineer to act on immediately?
A report that says “memory usage was high” without identifying which process, when it started, and what likely caused it does not meet the standard. A report that says “the Python process at PID 1842 began allocating ~500MB after the 04:07 database connection pool exhaustion event, and this corresponds to the period when response times degraded” does.
